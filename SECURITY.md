# Security Policy

## Reporting a vulnerability

**Do not open a public issue for a security vulnerability.** A public issue tells
everyone about the hole before it is closed.

Report privately using GitHub's
[private vulnerability reporting](https://github.com/itolgayazgi/whystack/security/advisories/new)
on this repository. If that is unavailable to you, contact the maintainer directly
rather than filing an issue.

Please include: what the issue is, how to reproduce it, and what an attacker could
achieve with it. A proof of concept helps, but a clear description is enough — do
not withhold a report because you have not built an exploit.

## What to expect

- **Acknowledgement within 72 hours.**
- An assessment and a fix plan, shared with you.
- Credit in the advisory when the fix ships, unless you prefer to stay anonymous.

Please give us a reasonable window to ship a fix before disclosing publicly.

## Supported versions

The project is pre-release. Only `main` is supported; there are no maintained
release branches yet.

## Scope

In scope: this repository's source code, its dependencies as configured here, and
the deployed API and client once they exist.

Out of scope: findings that require a compromised developer machine, social
engineering, or physical access. Reports generated by a scanner without a
demonstrated impact will be closed.

## What we do on our side

- Secrets are never committed. Placeholders in configuration are obviously fake.
- Tokens, JWTs, passwords and connection strings are never logged.
- Refresh tokens are stored hashed, never raw (ADR-0008).
- Dependency vulnerabilities are treated as build failures, not warnings — see
  the `Microsoft.OpenApi` pin in `apps/api/src/WhyStack.Api/WhyStack.Api.csproj`
  for a worked example.
